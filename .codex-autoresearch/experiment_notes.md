# Lumen HDR120 Autoresearch Experiment Notes

Best target: 106.25. Keep only experiments that beat best and preserve HDR, resolution, frame rate, codec support, and stability.

## Logging Protocol

- Record every measured experiment, including crash, tie, discard, and keep results.
- Each entry must include hypothesis, code change, measured result, and meaning for the next run.
- Treat failed experiments as closed evidence when they isolate an axis; do not repeat them unless the next test changes the topology or fixes an invalid measurement.
- Keep `results.tsv` as the compact ledger and this file as the rationale/history ledger.

## 1934 - MacDisplayKit 0d9255a - retry skylight stream creation

- Hypothesis: back-to-back HEVC then ProRes runtime probes sometimes fail because raw SkyLight returns nil while the prior stream is still being released; one short retry could stabilize suite measurement without changing encode quality.
- Change: wrapped `SLDisplayStreamCreateWithDispatchQueue` creation in a helper and retried once after 35 ms when the first create returned nil.
- Result: 106.25, discard because it tied best rather than beating it.
- Meaning: useful as measurement stabilization, but not a score improvement by itself. It does not address the HEVC 102-frame ceiling.

## 1935 - MacDisplayKit 49476a3 - complete low latency hevc frames

- Hypothesis: HEVC submits about 108 frames but only 102 arrive inside the 1 s probe; calling `VTCompressionSessionCompleteFrames` after each HEVC PQ submission might pull delayed encoder output into the probe window.
- Change: called `VTCompressionSessionCompleteFrames` synchronously after each HEVC PQ encode submission.
- Result: 71.33, discard. HEVC fell to 88 frames with 16 drop events.
- Meaning: synchronous completion blocks the submission path long enough to delay source release and create pending-frame pressure. This confirms immediate blocking drain is the wrong direction.

## 1936 - MacDisplayKit 8fa8808 - async drain low latency hevc

- Hypothesis: the completion drain might still help if it runs after enqueue on a separate queue, avoiding the source-release blockage from experiment 1935.
- Change: added a low-latency drain queue that calls `VTCompressionSessionCompleteFrames` asynchronously for HEVC PQ submissions, plus the one-shot SkyLight create retry to keep codec-suite probes stable.
- Result: 80.38, discard. HEVC produced 93 frames with 12 drop events; ProRes stayed healthy.
- Meaning: even non-blocking per-frame completion competes with normal VT scheduling and reduces HEVC throughput. The remaining 102-frame ceiling should not be attacked with per-frame `CompleteFrames`; the next useful direction is before VT submission or probe-window/output accounting, not force-draining every frame.

## 1937 - MacDisplayKit 8ccf5da - disable temporal hevc compression

- Hypothesis: HEVC PQ may be holding frames for temporal/reference compression even with frame reordering disabled; disabling temporal compression could lower output latency without lowering bitrate, HDR, resolution, or frame rate.
- Change: set `AllowTemporalCompression=false` for HEVC PQ sessions, plus the one-shot SkyLight create retry for suite stability.
- Result: 106.25, discard because it tied best. HEVC remained at 102 frames with 0 drops and callback latency stayed around 67 ms.
- Meaning: temporal/reference compression is not the 102-frame ceiling. VT still reports the same output cadence and latency, so the remaining bottleneck is not ordinary temporal compression holding.

## 1938 - MacDisplayKit 5f0fa73 - use unretained staging command buffers

- Hypothesis: staged HEVC conversion may be paying command-buffer resource-retain overhead; using Metal command buffers with unretained references could reduce staging overhead and push more than 108 submissions through the 1 s probe.
- Change: used `makeCommandBufferWithUnretainedReferences()` for staged encoder copies with fallback to the retained command-buffer path, plus the one-shot SkyLight create retry.
- Result: 106.25, discard because it tied best. HEVC stayed at 102 frames with 0 drops; Metal stage average was about 5.07 ms, not improved.
- Meaning: command-buffer resource-retain overhead is not the current limiter. The HEVC ceiling is downstream of staging or in fixed VT output latency.

## 1939 - MacDisplayKit 9a28d76 - batch low latency hevc drains

- Hypothesis: HEVC submits more frames than arrive inside the 1 s probe, but per-frame `VTCompressionSessionCompleteFrames` was too disruptive; batching completion drains at about 30 Hz might reduce fixed output latency without causing source-admission pressure.
- Change: added a low-latency drain queue and throttled HEVC PQ completion drains to roughly every 33 ms, plus the one-shot SkyLight create retry for codec-suite stability.
- Result: 104.04, discard. HEVC produced 101 frames with 1 drop event; average callback latency improved to about 49.6 ms, but progression fell and stability penalty appeared. ProRes stayed healthy at 112 frames.
- Meaning: periodic `CompleteFrames` can reduce measured callback latency, but it still costs enough throughput/stability to lose score. The useful signal is that output latency is movable, but the drain API is the wrong mechanism; future work should avoid completion-drain variants and look for output accounting/window alignment or pre-VT pipeline overlap that preserves submissions.

## 1940 - MacDisplayKit 3a6ee47 - inline skylight replay delivery

- Hypothesis: the SkyLight delivery path pays a per-frame unstructured `Task` allocation and actor hop even though callbacks are already serialized on a dedicated delivery queue; keeping replay state queue-confined could improve 120 fps source admission without changing encoder quality or codec behavior.
- Change: converted the private SkyLight replay state from an actor to a delivery-queue-confined class and called it synchronously from the SkyLight callback and replay timer.
- Result: 25.00, crash/inconclusive. Both HEVC and ProRes runtime probes failed before streaming with `SLDisplayStreamCreateWithDispatchQueue returned nil`.
- Meaning: this run does not evaluate the hot-path hypothesis. It re-confirms the suite-level SkyLight create race seen in 1934; the same hot-path change needs to be measured with the known one-shot create retry as measurement stabilization.

## 1941 - MacDisplayKit 3a6ee47+025cdae - inline skylight replay delivery with create retry

- Hypothesis: after stabilizing stream creation, removing the per-frame `Task` allocation and actor hop from SkyLight replay delivery should reduce source admission overhead enough to lift HEVC beyond the 101-102 frame ceiling.
- Change: kept the queue-confined replay state from 1940 and added the known one-shot SkyLight stream create retry solely to get a valid back-to-back HEVC/ProRes measurement.
- Result: 106.04, discard. HEVC produced 101 frames with 0 drops, 107 VT submissions, and about 67.3 ms average callback latency. ProRes produced 110 frames with 0 drops.
- Meaning: the SkyLight replay actor/Task hop is not the current limiter. It lowers encode queue wait noise, but HEVC still ends at 101 frames with fixed VT output latency. Future experiments should move away from source delivery concurrency overhead and toward either pre-VT overlap that increases submissions beyond 107-108 or a non-drain way to reduce VT output latency.

## 1942 - MacDisplayKit 8e1ed76 - inline staged vt submit

- Hypothesis: staged HEVC frames pay an extra dispatch hop from Metal completion into `submissionQueue` before `VTCompressionSessionEncodeFrame`; submitting directly from the Metal completion handler could preserve ordering while increasing VT submissions beyond the 107-108 ceiling.
- Change: called `submitToEncoder` directly from the Metal command-buffer completion handler for staged frames, and included the known one-shot SkyLight create retry as measurement stabilization.
- Result: 106.04, discard. HEVC stayed at 101 frames with 0 drops and 107 VT submissions; ProRes stayed healthy at 110 frames. HEVC Metal-stage timing ballooned to about 94 ms average because the metric now includes VT submission work inside the completion handler path.
- Meaning: the `submissionQueue` dispatch hop is not the limiter. The useful submit count does not increase, and folding VT submission into the Metal completion callback only shifts blocking into the Metal-stage timing bucket. Future work should not repeat inline staged submit; the remaining ceiling is the VT encode call/output latency itself or a deeper pre-VT cost reduction.

The partial HEVC leg suggests queue wait improved but did not lift the 107-submission / 101-output ceiling. Any future attempt to evaluate submit-path microstructure needs the known one-shot SkyLight create retry as measurement stabilization, but the retry alone has already tied best and should not be treated as the performance improvement.
