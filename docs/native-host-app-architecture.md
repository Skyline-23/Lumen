# Native Host Architecture

## Ownership

Rust is the single cross-platform authority. `lumen-engine` owns typed policy
and durable state; `lumen-host` owns the worker, servers, discovery, transport,
packetization, and application lifetime.

Rust owns:

- owner accounts, enrolled devices, credentials, and host identity;
- revisioned settings and the application catalog;
- launch admission, session negotiation, and control-channel routing;
- HTTPS account/device control, TLS 1.3 QUIC session control, native UDP media,
  mDNS, and IPv4/IPv6 UPnP;
- audio layout, video transport, HDR, bitrate, FEC, and fallback policy;
- process orchestration, recovery, and factory reset.

Platform adapters receive typed plans and return typed events. They do not own
a second copy of settings, protocol policy, or session state.

## macOS boundary

The SwiftUI application owns windows, menus, onboarding, permissions, and local
presentation state. Swift and Objective-C adapters own only macOS framework
boundaries:

- ScreenCaptureKit and CoreAudio capture;
- Metal and VideoToolbox resources;
- AppKit input and window operations;
- virtual-display creation and workspace restoration;
- the process bridge to the bundled Rust host worker.

Pixel buffers, audio buffers, textures, and encoded packets stay in their
native zero-copy path. Rust receives metadata and opaque resource handles.

The supported macOS build graph is generated by Tuist and contains no shared
C++ host target.

## Windows boundary

The Windows executable enters the Rust host. Rust calls DXGI, D3D11, Media
Foundation, WASAPI, Win32 input, service, and lifecycle APIs directly. C++ is
limited to the two process entry shims that transfer control to Rust and the
separately attributed ViGEmClient source required by that driver boundary.

- `lumen-host` owns Desktop Duplication, GPU color conversion, hardware encoder
  activation, audio capture, native input, and the Windows management UI;
- the selected output's Advanced Color state controls explicit BGRA8 or FP16
  `DuplicateOutput1` capture without a silent HDR downgrade;
- first-party C++ contains no protocol, policy, state, media, or UI engine.

The CMake build is Windows-only. It does not contain the retired C++ HTTP,
authentication, settings, application, network, or UPnP engines, or any legacy
RTSP/SDP/RTP/ENet session implementation.

Windows video encoding uses the operating system Media Foundation hardware
transforms. Lumen does not compile, link, or package FFmpeg, x264, x265, or a
vendor codec SDK. Opus remains the separately attributed native audio codec.

## Settings and application lifecycle

Native launch values are reconciled into the Rust settings authority before
the worker starts network or session services. Reconciliation preserves pending
client-authored values, and worker-restart values become effective only after
the new worker starts.

Application preparation, state, undo, detached, main-process, and indexed
server commands are orchestrated by Rust. Structured settings commands use
direct program-and-argument execution without a shell. Platform adapters handle
only process creation and privilege boundaries.

## Authentication and reset

The owner account is host-local. Enrolled clients authenticate with a
device-bound key, short-lived access token, and rotating revocable refresh
token. Streaming never reuses the owner password as a device credential.

Factory reset is an explicit local action. It removes authorities and product
state while retaining diagnostic logs needed to investigate failures.
