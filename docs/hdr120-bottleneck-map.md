## Failure Retrospective Protocol

Every discarded experiment has to answer "why did it fail?" before the next commit. The minimum review is:

| Question | Required evidence |
| --- | --- |
| Which boundary broke first? | Classify the first regression as source ingress, pending admission, Metal stage, VT submit/admission, VT output callback, Lumen forwarding, HDR validation, drop/stability accounting, or build/test failure. |
| Did it improve the intended stage? | Compare the targeted counters against the current baseline: HEVC frames, staged VT submissions, source count if available, Metal stage, VT encode call, VT callback latency, startup, queued/dropped frames, and drop events. |
| Did it preserve the contract? | Confirm HEVC Main10/PQ HDR, partial HDR requirement, target resolution, target fps, low latency, quality/bitrate policy, and ProRes suite health when applicable. |
| Why is it closed or still open? | State whether the axis is closed, invalid/inconclusive, or worth retrying only with a changed topology. |
| What should the next experiment avoid? | Record the concrete anti-repeat lesson in `.codex-autoresearch/experiment_notes.md`, not just in chat. |

Stability failures are usually not crashes in this campaign. If `DROP_EVENTS` is nonzero, trace the event source before interpreting the score. Current evidence points to MacDisplayKit source-frame drops when upstream admission gets ahead of VT submit/output and the pending window saturates; those drops enter the evaluator as stability penalties even when HDR and forwarding remain valid.

## Highest Priority Bottleneck

The best current explanation has shifted again:

1. For HEVC, single full-frame VT output was the old ceiling; the kept tile-record protocol now reaches 194 HDR tile records with zero drops.
2. For ProRes, experiment 2061-2063 recovers source cadence and reaches 134-141 HDR records with zero drops, so ProRes frame count is no longer the primary limiter.
3. The remaining HEVC gap is now 97 complete logical tile groups versus the 120 target. Tile geometry matters: column partitioning improved the previous 89-group audit.
4. The latest tail-latency evidence points at Metal conversion plus VT encode-call/output drain. HEVC VT encode-call P95 is about 21 ms in the diagnostic run, so the next useful structural experiment should reduce per-lane Metal/VT work or avoid requiring both lanes to complete each logical frame inside the same 8.33 ms cadence.
5. The first topology-contract probe supports the latter path: independent tile freshness can exceed 120 updates in the current stream, while strict same-group completion cannot. The next product-level bottleneck is the receiver contract: tile manifest, latest-tile composition, skew bounds, and visual-coherence rules.
6. The skew-bounded probes refine this: the current freshness stream needs a compositor policy, not just a metric rename. A plausible next contract is to accept latest-tile updates only under a one-frame or two-frame skew budget and explicitly account for held/dropped over-skew tile updates.
7. The skew distribution is lane-asymmetric, but 2076 shows that the stale lane follows enqueue order. That makes the next structural question narrower: strict groups are capped by ordered tile output cadence, while freshness composites need an order-aware bounded-staleness contract.
8. Sender-side paired output is the wrong layer for that contract. It improves skew distribution but causes drop amplification, so the next experiment should model receiver-side arbitration without blocking encoded tile output.
9. The first receiver-side model closes the naive latest-pending policy: two-frame arbitration is far below 120 updates, while three-frame arbitration jumps to 190 updates. The useful path is now phase-aware receiver scheduling or cadence normalization that can recover two-frame coherence without sender hot-path buffering.
10. The phase-normalized queue probe shows that two-frame coherence is present in the stream, but heavy in-probe arbitration destabilizes HEVC. This points toward offline trace analysis, lightweight metadata collection, or actual receiver-side implementation rather than more probe-side queue simulation.
11. The offline FIFO-front probe shows that phase reconstruction cannot just preserve queues and consume their fronts. The next useful measurement should expose tile arrival phase directly, or choose nearest-in-window candidates against a reconstructed presentation clock.
12. Source timestamp pairing also fails: nearest source-display-time matches equal strict groups. Receiver timebase reconstruction needs receiver-side arrival/callback phase, not only capture/display timestamps.
13. Arrival timestamp pairing also fails: nearest probe-drain arrival matches equal strict groups. The 2081 two-frame recovery came from stateful partial presentation updates, not from discovering more whole-tile pairs.
14. Bounded newest-pending presentation also fails. It drops the history that produced the 2081 signal, so queue-depth limiting by itself is not the stateful presentation contract.
15. Returning to Metal conversion with a fused no-scale 420 kernel also fails. The same staged BGRA topology still regresses to 94 complete groups and nonzero drops, so the next encoder-side experiment needs a different pacing/ownership topology rather than only fewer shader dispatches.
16. Moving color conversion into VT by submitting staged BGRA to a Main10 HEVC session fails harder. The explicit staged 420v10 path remains the better baseline; future VT/output work should not assume VT internal conversion is cheaper.
17. Staggering tile-lane PTS phases also fails. VT/output work should not assume a simple timestamp cadence offset can desynchronize hardware drain; the next encoder-side branch needs different session ownership, callback isolation, or a real client-side tile presentation contract.
18. Deferring callback-local HDR augmentation also fails. The VT callback latency bucket reflects downstream encoder/output drain more than local metadata wrapping work, so callback work placement alone is not enough.
19. Explicit per-frame duration also fails. The VT cadence problem is not fixed by making frame duration more concrete; this appears to add ordering or scheduler pressure rather than reducing drain jitter.
20. Sharing the tile-lane Metal command queue also fails. GPU completion ownership at the command-queue level does not explain the strict-group plateau; the bottleneck is deeper than simple Metal queue burst regularity.
21. Sharing the tile-lane VT output queue also fails. Output-handler serialization after VT callbacks does not change the ~70 ms callback latency regime or the 96 strict-group plateau, so the remaining encoder-side branch must alter VT session/drain ownership itself rather than only downstream queue ownership.
22. Skipping explicit VT session preparation also fails. The callback regime and strict-group plateau survive lazy session start, so the plateau is not just a `PrepareToEncodeFrames` lifecycle artifact.
23. Sharing the tile-lane VT session changes the callback regime but not the logical-group ceiling. One session reduced callback latency from ~70 ms to ~36 ms and doubled VT submissions, but strict complete groups stayed 96 and drops increased. The next useful session-topology branch should control admission/pacing around this lower-latency regime rather than treating raw shared-session throughput as a solution.
24. Matching shared-session hints to tile-record cadence preserves the lower-latency regime and recovers 97 complete groups, but drops remain. This keeps the shared-session branch alive only if the next experiment adds explicit in-flight/admission control.
25. A raw staging-slot cap is the wrong admission-control primitive. It keeps the lower callback-latency regime and removes drops, but turns pressure into `stagingSlotUnavailable` failures and collapses useful groups to 83.
26. Synchronous soft pacing is also the wrong primitive. Waiting inside the capture/encode path destabilizes ingress and can fail the HEVC runtime probe outright, so any remaining shared-session admission work must be non-blocking or moved out of the capture hot path.
27. Shared-session timestamp grouping is closed. Giving both tiles from one source frame the same PTS produces no useful HEVC output, so VT must see tile records as independently timestamped submissions unless a deeper encoded-tile protocol changes.
28. Shared-session non-blocking staged drain is also closed as a generic encoder-layer primitive. It preserves the lower callback-latency regime but does not improve strict group completion and breaks ProRes proxy, so remaining shared-session work must be codec-scoped or moved into an explicit receiver/client tile presentation contract.
29. Source-level pending-window reduction is also closed for shared-session admission. Lowering the HEVC tiled SkyLight pending window to 12 destabilizes the runtime probe, so the source ingress window should not be used as the next pacing knob.
30. Source-level pending-window expansion is closed too. Raising the HEVC tiled SkyLight pending window to 24 also destabilizes the runtime probe, so the shared-session branch should not continue with source-window sizing in either direction.
31. Shared-session zero frame-delay is closed. Forcing `MaxFrameDelayCount=0` on the shared HEVC tile processor fails both HEVC and ProRes runtime probes, so simple VT frame-delay knobs do not provide safe admission control around the lower-latency shared-session regime.
32. Probe hot-path drop-source classification is closed as implemented. Adding dropped-frame message buckets to the official runtime probe produced zero HEVC/ProRes frames, so future attribution should parse existing logs or use a separate non-scoring diagnostic runner.
33. The current clean measurement path can enter a no-frame startup failure before VT submission. Experiment 2104 converted that state into an explicit probe error and confirmed diagnostics stop at SkyLight/raw-private setup with 0 VT submissions. Future work should restore valid source delivery before further encoder or receiver tuning.
34. The no-frame source startup failure survives a probe-level capture restart. Experiment 2105 showed one stop/start retry still yields 0 HEVC/ProRes frames, startup -1.0 ms, and 0 VT submissions, so the next startup/source-delivery work must instrument or isolate the source callback/forwarding boundary rather than retrying the probe start sequence.
35. The current HEVC no-frame startup state is upstream of Lumen core-forwarding. Experiment 2106 exposed existing capture-session counters and showed `captureSessionIsRunning=true`, `captureSessionEmittedFrameCount=0`, `captureSessionProcessingFailureCount=0`, `captureSessionLastError=n/a`, and `VT_SUBMITTED_FRAME_COUNT=0`; the next boundary is raw-private SkyLight source callback emission or source-to-processor handoff.
36. The current no-frame startup state is upstream of the source-to-processor handoff. Experiment 2107 showed `SKYLIGHT_SOURCE_CALLBACK_COUNT=0` and `SKYLIGHT_SOURCE_TIMER_TICK_COUNT` around 1200 for both HEVC and ProRes, so the raw-private SkyLight display stream starts enough for the runtime timer to run but never invokes the frame callback during the metric window.
37. Private direct IOSurface fallback does not recover the current official probe. Experiment 2108 forced the probe away from raw SkyLight and both codec probes failed immediately with a nonzero private-direct warmup status, so the next startup experiment should stay on raw SkyLight callback creation/start parameters or introduce a different fallback backend.
38. Raw SkyLight rect geometry properties do not recover callback delivery. Experiment 2109 added source rect, destination rect, and preserveAspectRatio properties only in the probe, but both codec probes still produced 0 frames and 0 VT submissions, so the missing callback is not explained by absent CGDisplayStream-style rect keys.
39. Nil raw SkyLight stream properties do not recover callback delivery either. Experiment 2110 removed all stream properties only in the probe and still failed the official metric, including HEVC runtime failure and ProRes 0 frames/0 VT submissions, so the next property-profile work should not jump to nil-profile extremes.
40. The existing private probe's rectless-minimal property shape does not recover callback delivery. Experiment 2111 set `showCursor=false`, `minimumFrameTime=0`, `queueDepth=3`, `preserveAspectRatio=false`, and no rects, but the official runtime probe still failed with no ProRes frames or VT submissions.
41. Strong callback-owner retention does not recover raw SkyLight callback delivery. Experiment 2112 removed the weak-owner check from the raw stream callback path, but both codec probes still produced 0 frames and 0 VT submissions, so callback block owner lifetime is not the first failing boundary.
42. A generic public CGDisplayStream encoded-source fallback does not recover the official probe. Experiment 2113 wrapped `MDKCaptureSessionFactory(.cgDisplayStream)` as an encoded source and forced it from the probe, but both HEVC and ProRes failed before producing usable frame diagnostics.
43. Starting a raw-created stream through public `CGDisplayStreamStart` does not recover callback delivery. Experiment 2114 kept `SLDisplayStreamCreateWithDispatchQueue` but replaced `SLDisplayStreamStart`, and HEVC still produced 0 frames/0 VT submissions while ProRes failed the runtime probe.
44. Raw SkyLight create/start lifecycle is not the failing boundary. Experiment 2115 showed `SLDisplayStreamCreateWithDispatchQueue` and `SLDisplayStreamStart` are present, stream creation succeeds, `SLDisplayStreamStart` returns 0, size/pixel format/properties match the request, and HEVC still produces 0 frames and 0 VT submissions.
45. Main-queue raw SkyLight callback targeting is not viable. Experiment 2116 forced the raw stream to use `dispatch_get_main_queue()` only in the probe, but the official metric hung without parseable output rather than delivering source frames.
46. Raw SkyLight preflight benchmarking changes the operating state but is not a keepable fix. Experiment 2117 ran a short standalone raw benchmark before encoded capture; it restored HEVC/ProRes frame delivery but only reached 95 HEVC complete groups with 6 drops and 98.00 total score.
47. Minimal raw SkyLight warm-up is enough to unblock startup but not enough to improve the target. Experiment 2118 reduced preflight to 10ms and surfaced 2 callbacks / 1 complete warm-up frame, restoring delivery but landing at 97 HEVC complete groups with 8 drops and 94.00 total score.

The target is now to make the HEVC tile-stream contract rigorous enough for the product: either clients and probes must explicitly consume independent encoded tile records as tile substreams, or the encoder topology must deliver 120 complete logical tile groups without treating valid substreams as drops.

## Next Structural Directions

- Tile semantics branch: add a first-class tile-stream manifest/reassembly contract so the receiver can map each encoded tile record to frame group, tile index, region, and lane without requiring the encoder to hold back records for group completion.
- Tile freshness branch: define a skew-bounded partial-freshness contract where clients can update a tile as soon as its substream arrives after all tiles are primed, instead of waiting for all lanes from the same frame group. The first version should expose tile age/skew metadata and hold/drop over-budget updates rather than blindly presenting every latest tile.
- Ordered-output branch: receiver-side temporal arbitration is now preferred over sender-side paired emission, but the naive latest-pending budget, heavy probe-side queues, FIFO-front offline pairing, source-time nearest matching, arrival-time nearest matching, and bounded newest-pending state are closed. The next useful version should either reproduce 2081's exact stateful update semantics with less hot-path work, or move to an actual client-side presentation contract instead of more probe-only pairing searches.
- Logical-group throughput branch: if the product requires complete frames before delivery, redesign HEVC lane scheduling so both tile lanes produce at least 120 complete groups/s; do not reintroduce complete-group gating on the current queue because prior gating collapsed throughput.
- Client contract branch: define whether Android/general clients consume independent HEVC substreams directly, reassemble them, or require a logical-frame manifest beside each tile record.
- Lane-count branch is mostly closed for direct scaling: 3 lanes increased record count but made logical groups worse. Prefer 2-lane scheduling/region work before any higher lane count.
- Lane-order branch is closed for simple rotation: reordering existing lane work caused incomplete groups. Per-lane queue isolation also tied the current 97-group best without moving VT submission/output, so prefer reducing per-lane work or changing the encoded tile contract rather than rescheduling the same work.
- Tile-width alignment branch is closed for simple asymmetric CTU rounding: equal half-width columns remain the best measured geometry.
- ProRes catch-up branch is kept and should not be widened unless new evidence shows drop-free ProRes overproduction hurts battery or latency.
- HEVC admission branch is lower priority now: keep the 2058 tile path unless a new change regresses HDR, drops, or Android-required HEVC Main10 support.
- VT cadence branch: simple timestamp phase staggering is closed. If this branch continues, change VT session topology or callback/output ownership rather than only shifting presentation timestamps.
- Callback ownership branch: simple deferral of HEVC HDR augmentation and simple tile-lane output-queue sharing are closed. Future callback work must change VT session/drain ownership more structurally than moving sample-buffer wrapping or output delivery onto a different serial queue.
- VT timing-hint branch: explicit per-frame duration is closed. Keep session-level duration hints and avoid more timestamp/duration-only tweaks unless coupled to a different session topology.
- VT lifecycle branch: skipping `VTCompressionSessionPrepareToEncodeFrames` is closed. Do not retry lazy session start without changing session ownership or output contract.
- VT session topology branch: one shared tile VT session is not keepable as-is, but it proves session topology affects callback latency. Tile-record cadence hints alone are also insufficient, hard staging-slot caps create admission failures, synchronous pending waits destabilize capture, same-PTS grouping kills output, generic non-blocking staged-submission drain breaks ProRes while regressing strict groups, source pending-window sizing fails in both directions, and `MaxFrameDelayCount=0` fails both runtime probes. If this branch continues, it must use a genuinely new per-codec lifecycle model, or it should move to a receiver/client presentation contract around the shared-session shape instead of another admission/pacing knob.
- Metal ownership branch: shared tile-lane command queue is closed. Do not repeat queue-sharing without changing VT session ownership or presentation contract.
- Probe instrumentation branch: do not add new counters directly to the official runtime probe unless the measurement target is otherwise unchanged and a baseline sanity run confirms frame delivery. Prefer log-parser-only diagnostics for attribution work.
- Startup/source-delivery branch: the immediate blocker is now a zero-frame startup path on the clean tree. Before more VT/session/presentation tuning, isolate why SkyLight/raw-private capture starts but no forwarded frames reach VT in the official probe. Experiment 2105 closes a generic stop/start retry, experiment 2106 closes Lumen core-forwarding as the first explanation for HEVC zero frames, experiment 2107 closes source-to-processor handoff as the next boundary because the raw SkyLight callback count is zero while the runtime timer ticks, experiment 2108 closes private direct IOSurface fallback in the current probe, experiment 2109 closes rect/preserveAspectRatio property additions, experiment 2110 closes nil stream properties, experiment 2111 closes the private-probe rectless-minimal profile, experiment 2112 closes weak callback-owner retention, experiment 2113 closes a generic public CGDisplayStream encoded-source wrapper, experiment 2114 closes public `CGDisplayStreamStart` as a raw-stream startup substitute, experiment 2115 closes raw create/start availability/status diagnostics, and experiment 2116 closes main-queue callback delivery. Experiment 2117 shows a standalone raw benchmark before encoded capture can restore delivery, and experiment 2118 shows the unblock can shrink to a 10ms warm-up with 2 callbacks / 1 complete frame. The next useful experiment should not be another duration-only warm-up; it should combine the minimal source unblock with a different downstream admission/drop strategy, or identify the private side effect without producing extra warm-up frames that perturb source/VT pressure.
